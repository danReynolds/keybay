Cross-platform.OS-backed.

Cross-platform secret storage for Dart and Flutter, plus a five-command CLI for local processes. Keybay applies one documented, OS-backed storage policy for each supported runtime and fails closed when it cannot.

The child receives ordinary environment variables. Keybay replaces itself with that process; no resident Keybay wrapper remains.

Automatic platform policy

One fixed path per runtime.

No backend flags and no plaintext key-on-disk fallback. Keybay reports the resolved scheme and observed security level instead of inferring hardware from an OS name.

Surface and runtimeSecrets live inProtection
SDK · iOSData Protection Keychain itemsAfterFirstUnlockThisDeviceOnly, non-synchronizing; hardware backing not attested
SDK · entitled macOS appData Protection Keychain itemsThe same fixed item policy; hardware backing not attested
SDK/CLI · unentitled macOSAuthenticated encrypted file32-byte store key in the login Keychain; login-bound
SDK · Android 12+Authenticated encrypted fileStore key wrapped by Android Keystore; StrongBox requested and actual level inspected
SDK/CLI · Linux desktopAuthenticated encrypted file32-byte store key in an unlocked Secret Service provider; login-bound

Windows is unsupported. Headless deployments have no supported backend or availability contract. Keybay never substitutes plaintext storage or a plaintext store key beside the encrypted container.

Two interfaces

Read directly—or launch any process.

The SDK exposes the bytes-first store to Dart and Flutter. The CLI resolves committed references for applications that already read environment variables.

Dart / Flutter SDK

Read and write directly.

One appId names the logical store; the runtime selects the fixed platform path. Values are bytes at the core, with string conveniences.

Dart

import 'package:keybay/keybay.dart';

final secrets = SecretStorage(
  appId: 'com.example.myapp',
);

await secrets.writeString('api-token', tokenFromOAuth);
final stored = await secrets.readString('api-token');

macOS / Linux CLI

Five commands. No resident service.

A committed manifest defines the contract; each developer supplies project-qualified values to one local per-user store. Namespaces organize identity, not access control.

Security

No usable store, no secret access.

Keybay is designed around explicit guarantees and explicit limits. It does not turn unavailable platform protection into a hidden fallback.

Fail closed
Locked, missing, invalidated, corrupt, and unsupported states return typed failures. No plaintext store-key fallback.
Authenticated file path
XChaCha20-Poly1305, key commitment, atomic 0600 writes, fsync, and cross-process locking.
Native Apple path
Data Protection Keychain items use a fixed, device-bound, non-synchronizing accessibility policy.
Report only what is known
Android hardware level comes from its wrapping key's provider. Apple native items report no hardware level because Keybay cannot attest one.
Small review surface
One exact-pinned third-party runtime dependency, known-answer vectors, and genuine platform-store integration paths.
Honest boundary
Same-user malware, process memory, child environments, rollback, and root remain outside the threat model.

Scope

When Keybay is the right tool.

Choose Keybay when each developer owns local values and you want a small Dart API or run-scoped injection with no hosted control plane. It is deliberately not a team secrets platform.

Full comparison →
  • SecretSpec ↗Choose it for profiles, provider selection, fallback chains, and multi-environment declarations.
  • 1Password CLI ↗Choose it for shared vaults, administration, service accounts, and team workflows.
  • dotenvx ↗Choose it when encrypted environment files should travel with a project or deployment.
  • flutter_secure_storage ↗Choose it for Flutter-specific controls or platforms Keybay does not support.
